# Phase 3: Render Pipeline Initialization - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Set up Blender (headless) to ingest pre-exported JSON files generated by the math engine and spawn default geometric objects (spheres/icospheres) exactly at the 3D coordinates provided. The renderer must successfully place geometry at `{x,y,z}` coordinates from the primes and fractal endpoints. Shaders, materials, animation, and export are out of scope for this phase.

</domain>

<decisions>
## Implementation Decisions

### Renderer Choice
- **D-01:** Use **Blender (headless)** as the rendering backend — run via `blender --background --python render.py`. Blender is Python-scriptable, produces broadcast-quality output, and directly supports the 4K offline render pipeline planned in Phases 4–5.

### Data Ingestion
- **D-02:** **Pre-export JSON files to disk** — the math engine saves `.json` output files; the Blender script reads them directly at render time. No live server dependency during rendering. The engine server is used to generate the data, then the files are persisted to a known path before the renderer is invoked.

### Geometry Representation
- **D-03:** Place a **sphere (icosphere)** at each 3D coordinate. Icospheres are smooth, geometrically clean, and scale naturally to represent mathematical point-data at any density. Default Blender icosphere primitive is appropriate.

### Renderer Invocation
- **D-04 (Agent's Discretion):** How the render pipeline is triggered (CLI script, Python subprocess, API endpoint, or Makefile target) is left to the planner. The priority is clean separation between the math engine and renderer — the invocation mechanism should not couple them tightly.

</decisions>
